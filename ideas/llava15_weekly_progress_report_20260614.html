LLaVA Weekly Progress Report

Updated June 14, 2026. This report summarizes the cleaned spreadsheet rows 163-179 for the JAX LLaVA reproduction block. Benchmark values come from PaliGemma-baseline-cleaned-20260608; run identity, config checks, image/text RMS ratios, and speed come from W&B summaries or local output.log text. No benchmark datasets or checkpoints were scanned.

Core Message

More SFT helps; late fusion does not yet.

Best Current Direction

Standard LLaVA-style fusion, stronger SFT recipe.

Main Risk

Late-fusion scale can make image tokens ineffective.

Speed Finding

Late fusion was slower in the measured runs.

Executive Readout

  • Beam search is not a clean global win. Row 164 beam search improves VisWiz strongly and helps several QA-style metrics, but drops MME from row 163's 1064 / 1243 to 976 / 1189. VQAv2 in row 164 is capped to about 10%, so that cell should be read with caution.
  • Training the vision encoder in stage 1 did not help. Row 166 collapses with the original stage-1 vision LR. Row 167 manually lowers LR and recovers, but still does not beat the row 165 laion-aes baseline overall.
  • The stage-2 connector can tolerate and benefits from a larger LR. Row 168 with connector LR 5e-4 is much stronger than row 166 and is competitive with or better than the row 165 laion baseline on many metrics.
  • Late fusion is very scale-sensitive. Rows 175-177 show that fixed bad scales or automatic stat matching can dramatically change behavior. The prompt-stat scale is the best of the layer-13 group on OCR/Text/VisWiz-style tasks, but all layer-13 rows are still far below the non-late-fusion baseline.
  • Late fusion has not shown a quality or speed advantage. Row 165 beats rows 177 and 179 on most benchmarks and training loss. Stage-2 median speed is about 3.57 steps/s for row 165, versus 0.87 for row 177 and 1.14 for row 179.
  • More SFT data/compute helps the LLaVA-style path. Row 171 improves VQAv2, TextVQA, MMBench, OCRBench, VisWiz, ScienceQA, GQA, and loss through 4x batch plus sqrt-style LR scaling. Row 172 improves MME, CountBench, ScienceQA, and several robustness probes by training stage 2 much longer.

3. Late-Fusion Init Scale: Rows 175-177

Layer-13 late fusion depends strongly on image-token scale. Fixed scale 1 leaves the model weak on most text/QA benchmarks; fixed scale 100 changes the image/text RMS ratio by two orders of magnitude and still underperforms. Prompt-stat matching is the best of these three on TextVQA, OCRBench, CountBench, VisWiz, and loss, but the absolute results remain poor. The practical message is that late fusion needs a principled init scale; otherwise image tokens can be ignored or over-amplified.

5. More SFT For LLaVA-Style Fusion: Rows 170-172

The cc12m/160-token LLaVA-style path responds well to more SFT effective compute. Row 171 increases stage-2 batch from 128 to 512 and scales LR upward; it gives the best VQAv2, TextVQA, MMBench, POPE, OCRBench, VisWiz, ScienceQA, GQA, train acc, and train loss in this trio. Row 172 trains stage 2 for 15000 steps and gives the best MME, CountBench, SEED, ScienceQA, and a stronger overall long-training signal than row 170.

Recommended Next Steps

  • Keep the standard LLaVA-style fusion path as the mainline and spend next runs on stronger SFT schedules/data rather than late-fusion complexity.
  • Use the row 171 recipe as the short-horizon performance baseline when budget allows larger batch and sqrt LR scaling; use row 172 when testing longer SFT convergence.
  • If late fusion is revisited, first solve the image/text scale initialization problem and measure early-stage image-token usage before launching full benchmark sweeps.
  • Do not make beam search the default table metric. Keep greedy as the main comparable protocol and log beam as an eval ablation for tasks such as VisWiz.
