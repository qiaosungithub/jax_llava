Multi-Trunk VLM — 3-Trunk Baseline and 4-Trunk Extension

A research proposal: organizing a VLM as a sequence of "trunks" with explicit causal ordering, and re-encoding the image after the question for question-aware vision features

Context for this document. This is the next round of brainstorming, building on the broader research direction sketched in brainstorm/01 (LLM-agnostic vision encoder) and brainstorm/03 (vision encoder pretraining survey). The high-level goal remains the same: train a strong vision encoder using a (preferably frozen) LLM as supervisor, with the vision encoder as the deliverable. This document refines the architectural picture into a "trunk-based" framing and proposes a 4-trunk extension where the image is re-encoded after the question is seen.

The high-level goal (recap)

Goal: Use a (preferably frozen) LLM as supervisor to train a Vision Encoder. The vision encoder is always trained and is the main object of the training procedure. We want a recipe similar in spirit to PaliGemma but with the vision encoder as the deliverable, optimized for the supervision signal a frozen LLM can provide.

The 3-trunk baseline (PaliGemma-style)

We can view a PaliGemma-style VLM as a sequence of three trunks:

  • Trunk 1 — Image: image patches encoded by the vision encoder.
  • Trunk 2 — Text prompt (question): the user's question, embedded into the LLM's token space.
  • Trunk 3 — Text response (answer): the answer, generated autoregressively.

Inter-trunk attention rule: causal between trunks. A trunk can only see itself or preceding trunks. Intra-trunk attention: bidirectional for vision (default), causal for text (default).

Why this baseline matters

This is essentially the LLaVA / PaliGemma layout, framed as trunks. Two design knobs are exposed:

  1. Inter-trunk attention: strict causal — each trunk only sees itself + preceding trunks. This matches the "input → context → output" flow.
  2. Intra-trunk attention: configurable per trunk. Vision is naturally 2D so default is bidirectional; text is sequential so default is causal.

Calling out these knobs explicitly makes it easy to reason about the next extension.

The 4-trunk extension — re-encode the image after the question

The new idea: Add a fourth trunk by re-encoding the same image a second time, this time after the model has seen the question. The sequence becomes:
  1. Trunk 1: Image (first encoding, sees nothing)
  2. Trunk 2: Text prompt (the question, sees T1)
  3. Trunk 3: Image again (second encoding, sees T1 + T2 — i.e., the question is in context)
  4. Trunk 4: Text response (the answer, sees T1, T2, T3)
The second image trunk lets the vision encoder produce question-aware features — features that have been computed with the question already in context.

Figure 2: The 4-trunk extension. Trunks 1 and 3 process the same image — Trunk 1 produces unconditional visual features; Trunk 3 produces question-aware features (because by the time Trunk 3 runs, the question has already been seen via the causal attention). Both encodings flow into the answer.

What's new in the 4-trunk version

The single conceptual innovation: the same image is encoded twice, with the second encoding conditioned on having seen the question.

Why this is interesting

In a standard PaliGemma-style 3-trunk model, the image encoder is question-agnostic: it produces the same features regardless of what's asked. The LLM has to do all the question-relevant feature selection on the language side, working from a fixed visual representation.

The 4-trunk version lets the vision encoder itself produce question-aware features. Trunk 3's encoder has access to:

  • The original image (via Trunk 1, plus the same pixel input)
  • The question (via Trunk 2)

So Trunk 3 can ask: "given the question, which visual features matter most? Which regions should I emphasize? Which fine-grained details should I extract that I didn't bother with the first time?"

This mirrors how humans look at images for VQA: a quick first glance to understand the scene, then read the question, then a focused second look to extract the answer-relevant information.

What it could enable for vision encoder training

Recall the high-level goal: train a strong vision encoder using a frozen LLM. The 4-trunk design changes what "strong" means for the encoder:

  1. Trunk 1's encoder learns general-purpose features (no question available). This is closest to a CLIP/SigLIP-style "summarize the image" encoding.
  2. Trunk 3's encoder learns conditional, attention-driven features (question available). This is closer to "look at the image with a goal in mind."

If the same encoder weights handle both passes, the encoder must learn both behaviors — general visual understanding AND question-conditioned focus. This is a richer learning signal than either alone.

Alternative: use two separate encoders for Trunk 1 and Trunk 3, with different weights. Then Trunk 1's encoder is a pure feature extractor (CLIP-like) and Trunk 3's encoder is specifically a question-conditioned reader.

Open design questions

Several knobs to set before this becomes concrete:

QuestionOptionsTradeoff
Same encoder weights for T1 and T3? (a) Shared (single encoder, used twice) (b) Separate (two encoders) Shared = parameter-efficient + forces unified behavior. Separate = more capacity, easier to specialize.
What does T3 actually attend to? (a) T3 self-attention only sees T3, with cross-attention to T1+T2 (b) T3 is in the same Transformer stack as T1+T2 (one big sequence) (a) is closer to a "question-conditioned vision encoder." (b) is simpler — just a longer sequence. Choice affects implementation a lot.
Does T3 produce raw image features (like T1) or summary tokens (like Q-Former)? (a) Same N tokens as T1 (b) Compressed K queries (Q-Former-like) (a) gives T4 maximum information. (b) gives T4 a focused "summary for this question" — could be a good information bottleneck.
How much extra compute does T3 add? ~1× the vision encoder cost (for the second pass) Doubles vision encoder forward pass cost. May be worth it if the encoder is small relative to the LLM.
What if the question is multi-turn? (a) Re-encode after every question (T3, T5, T7, ...) (b) Re-encode just once (a) is more powerful but expensive. (b) is the cheap default.
How does this interact with the "LLM-agnostic encoder" goal? The encoder still gets gradients via the frozen LLM. T3 specifically learns to encode for "what this LLM will need next" — this could leak LLM-specific bias. Could mitigate via multi-LLM training (from brainstorm/01): T3's encoder must produce features that satisfy multiple frozen LLMs.

Connections to existing work

The 4-trunk pattern is reminiscent of (but not identical to) several existing ideas:

  • Two-pass attention in classical VQA (e.g., Show, Attend and Tell; Bottom-Up Top-Down) — model produces attention over image features conditioned on the question. The 4-trunk design generalizes this to a full re-encoding rather than just attention reweighting.
  • Q-Former (BLIP-2; see related/09) — learnable queries pull task-relevant information from frozen image features. The 4-trunk Trunk 3 plays a similar role, but the queries come from the actual question text rather than from learned vectors.
  • Cross-attention insertion (Flamingo) — frozen LLM with trainable cross-attention layers that look at image features. The 4-trunk design is the inverse: trainable image encoder that looks at frozen text features.
  • "Thinking with Images" (V*, Visual Sketchpad — see deepseek_vision/01) — model iteratively re-examines the image at higher resolution. 4-trunk is a structural single-pass version of "look again" — the second look is built into the architecture rather than triggered at test time.
  • Reasoning-conditioned encoding in some recent VLMs (e.g., InternVL re-encoding crops) — similar spirit, different mechanism.

What's distinctive about the 4-trunk framing: it's an architectural commitment to question-conditioned vision encoding, with explicit causal ordering between trunks. Most prior work either (a) does this implicitly via attention without re-encoding pixels, or (b) does it at test time via tool use rather than as part of training.

Why this serves the high-level goal

Recall the high-level goal: train a strong vision encoder using a frozen LLM as supervisor. The 4-trunk extension serves this goal in two ways:

  1. Richer supervision signal. The vision encoder now receives gradients via two paths: (a) Trunk 1's contribution to the answer (general features matter), and (b) Trunk 3's contribution to the answer (question-conditioned features matter). The encoder must learn to do both.
  2. Forces the encoder to be task-aware. By being given the question before encoding, the encoder learns to produce features that are useful given a goal, not just "what's in this image." This is closer to how vision is used in practice — almost always as input to some downstream task.

The risk: Trunk 3's encoding may "overfit" to the questions seen in training. Mitigations: diverse questions per image, multi-LLM training (from brainstorm/01), and possibly periodically training Trunk 3 with the question masked to preserve generality.

This document is the architectural concrete next step: given the goal (brainstorm/01) and the literature (brainstorm/03), here's a specific design (multi-trunk) that operationalizes the goal.

The 4-trunk design takes the standard PaliGemma-style "Image → Prompt → Response" layout and inserts a second image-encoding trunk after the prompt, letting the vision encoder produce question-aware features. The proposal is structurally simple but has substantial design questions to resolve (shared vs separate encoders, what T3 attends to, raw vs compressed output). The goal is to give the vision encoder richer supervision — both unconditional (Trunk 1) and conditional (Trunk 3) — while keeping the LLM frozen.
