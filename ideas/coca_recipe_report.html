TL;DR

一句话理解 CoCa

CoCa 不是简单“CLIP + captioning 两阶段”，而是 单阶段、同一计算图、同一次 forward 同时优化 L_contrastive 和 L_caption： 下半段 decoder 做 text-only 表征，上半段 decoder 做 image-text 融合与自回归生成。

  • 你现在的 txt_feature_layer + 后层融合 思路与 CoCa 的 decoupled decoder 高度同构。
  • CoCa 的关键不是“投敌 CLIP”，而是用 contrastive 约束保全局对齐、用 captioning 保细粒度语义。
Architecture

模型结构 recipe

1) 解耦 decoder（核心）

  • Decoder 前半层（n_uni）不看图像，只做因果自注意力，输出 unimodal text embedding。
  • Decoder 后半层（n_multi）继续因果自注意力，并 cross-attend image tokens，输出 multimodal 表征。
  • 默认 n_uni = n_multi（“一半一半”）。

2) 双损失同训

  • L = λ_con * L_con + λ_cap * L_cap。
  • L_con 用 image embedding 与 text embedding 做对比学习。
  • L_cap 用 multimodal decoder 输出做 teacher-forcing 自回归预测。

3) 任务特定 pooler

  • contrastive 分支用单 query pooler（全局 embedding）。
  • generative 分支用长序列 query（默认 256）保细粒度视觉 token 供 decoder 使用。
  • 小规模 ablation 中 “cascade pooler” 略优于并行 pooler。

Ablation Findings

对你最有用的 ablation 结论

1) 损失配比

  • λ_cap:λ_con = 2:1 在他们的小规模消融里优于 1:1 / 1:2。
  • 结论：captioning loss 不只是帮生成，也改善对齐和 VQA。

2) uni / multi 层数平衡

  • uni 层太少会伤 zero-shot 对齐；multi 层太少会伤多模态推理。
  • “一半一半”是可行平衡点（与你的 split 设计吻合）。

3) 对比文本 embedding

  • 用单个可学习 [CLS] 做对比文本表征，比“[CLS]+全部 token 聚合”更稳。
  • 解释：减少 contrastive 与 captioning 目标在文本侧的干扰。

4) pooler 序列长度

  • generative 分支保留长 token（query=256）对多模态理解更有利。
  • 如果把 generative query 缩得太短，会明显伤 VQA。

For Your Project

映射到你当前架构的可执行建议

  1. 保留 split 架构：你的 txt_feature_layer 本质就是 CoCa 的 n_uni/n_multi。优先扫 1/3、1/2、2/3 切分点。
  2. 不要只靠 NTP：加一个低权重 image-text contrastive 分支（建议从 λ_con = 0.25~0.5 * λ_vlm 起）。
  3. captioning 权重要偏大：CoCa 结论偏向 λ_cap > λ_con；你可用 vlm:con = 2:1 作为初始点。
  4. 视觉 token 保持足够长：若你要做 OCR/grounding，K 不建议过早降到很小；先保 256，再做 token budget 课程学习。
  5. 文本对比头简化：可仿 CoCa 用单 CLS/单 query 作为 text embedding，减轻生成目标冲突。
  6. 冻结策略：先冻结前层稳态训练，再小 LR staged unfreeze；这和 CoCa 的“先保证双目标兼容”逻辑一致。
What CoCa Does NOT Say

你需要谨慎外推的地方

  • CoCa 使用的是极大规模数据与 batch；小规模下超参最优点可能偏移。
  • CoCa 的 tokenizer/数据清洗/去重流程很重，复现时数据质量对结果影响非常大。
  • 他们不是“冻结 LLM”范式，而是从头训 encoder-decoder；和你“加载预训练 LM 并部分冻结”有分布差。
